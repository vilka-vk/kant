# Directive: Design System Sync

## Goal
Выстроить и поддерживать связь между дизайн-системой в Figma
и кодовой базой проекта. Агент выступает мостом: извлекает
токены, компоненты и стили из Figma, транслирует их в код,
а при изменениях в любую сторону — синхронизирует.

## Role
Design Systems Engineer. Владеет Figma Plugin API,
CSS-токенами, компонентным подходом. Понимает, что
дизайн-система — это контракт между дизайном и разработкой.

## Input
1. Ссылка на Figma-файл (или file key)
2. Кодовая база проекта (любой стек)
3. (Опционально) Существующие токены, компоненты, стили

## Фаза 1: Аудит Figma-файла

### 1.1 Инвентаризация
Используй `mcp__figma__get_metadata` и `mcp__figma__get_design_context`
чтобы собрать:
- **Страницы** — названия и назначение (DS, макеты, архив)
- **Компоненты** — component sets, их варианты (props, states)
- **Стили** — текстовые стили (шрифт, размер, weight, line-height)
- **Переменные** — коллекции цветов, spacing, radius, теней
- **Иконки** — наборы иконок (если есть как компоненты)

### 1.2 Карта дизайн-системы
Зафиксируй результат в **проектную директиву** `design_system.md`:

```
| Категория     | Кол-во | Источник в Figma        |
|---------------|--------|-------------------------|
| Цвета         | N      | Variable collection "X" |
| Spacing       | N      | Variable collection "Y" |
| Radius        | N      | Variable collection "Z" |
| Text styles   | N      | Published styles        |
| Компоненты    | N      | Страница "Components"   |
| Иконки        | N      | Страница "Icons"        |
```

Эта директива — проектоспецифичная, содержит file key, node-id,
маппинг компонентов на код. Она создаётся один раз и обновляется.

## Фаза 2: Извлечение токенов

### 2.1 Цвета
Из переменных Figma (Variable collections) или из стилей:
- Собери все цветовые переменные с группировкой
  (brand, semantic, background, text, stroke, accent)
- Транслируй в формат проекта:
  - CSS Custom Properties (`--color-brand: #FF0032`)
  - SCSS/Less переменные
  - Tailwind config
  - JS/TS объект
  - или другой формат, принятый в кодовой базе

### 2.2 Типографика
Из текстовых стилей Figma:
- Font family, weight, size, line-height, letter-spacing
- Проверь доступность шрифтов: Google Fonts / системные /
  кастомные (нужна загрузка и подключение @font-face)
- Если шрифт кастомный — извлеки URL из исходных CSS/HTML,
  скачай, сконвертируй если нужно (woff2→ttf для десктопа)

### 2.3 Spacing, Radius, Shadows
- Spacing-шкала (4px, 8px, 12px, 16px, 24px, 32px...)
- Border-radius (xs, s, m, l, xl, full)
- Box-shadow (s, m, l)

### 2.4 Выходной файл
Создай единый файл токенов (e.g. `tokens.css`, `theme.ts`,
`tailwind.config.js` — по конвенциям проекта).

## Фаза 3: Компоненты

### 3.1 Приоритизация
Не создавай все компоненты сразу. Определи порядок:
1. **Атомы** — Button, Input, Checkbox, Radio, Toggle, Badge, Tag
2. **Молекулы** — Select, Alert, Tooltip, Modal, Tabs
3. **Организмы** — Table, Sidebar, Header, Card
4. **Шаблоны** — Layout, Page Shell

### 3.2 Для каждого компонента
1. Получи дизайн: `get_design_context` с node-id компонента
2. Определи API: props, variants, states, slots
3. Реализуй в стеке проекта (React, Vue, Svelte, HTML...)
4. Используй ТОЛЬКО токены из Фазы 2, не хардкодь значения
5. Покрой все состояния из Figma: default, hover, active,
   disabled, focused, loading, error

### 3.3 Code Connect (если поддерживается)
Если в проекте настроен Figma Code Connect:
- Создай маппинг `Component.figma.tsx` рядом с компонентом
- Связь: Figma node-id → React/Vue component + prop mapping
- Это позволяет Figma Dev Mode показывать актуальный код

## Фаза 4: Макеты (Screens)

### 4.1 Сборка экранов
При создании новых экранов или воспроизведении макетов:
- Собирай ТОЛЬКО из существующих компонентов и токенов
- Если нужен элемент, которого нет в DS — сначала создай
  компонент, потом используй
- Layout строй на токенах spacing, не на магических числах

### 4.2 Figma → Code
При переносе макета из Figma в код:
1. `get_design_context` — получить структуру и скриншот
2. Декомпозировать на существующие компоненты
3. Адаптировать сгенерированный код к стеку проекта
4. Заменить Tailwind/inline стили на токены проекта

### 4.3 Code → Figma
При добавлении экранов в Figma программно (`use_figma`):
- Используй существующие компоненты через `createInstance()`
- Привязывай переменные через `setBoundVariableForPaint()`
- Применяй текстовые стили через `textStyleId`
- Не создавай "сырые" фреймы там, где есть компонент

## Фаза 5: Синхронизация

### 5.1 Обнаружение дрифта
При каждом обращении к дизайн-системе проверяй:
- Совпадают ли токены в коде с переменными в Figma?
- Не появились ли новые компоненты/варианты в Figma?
- Нет ли в коде хардкоженных значений вместо токенов?

### 5.2 Обновление токенов
Если источник изменился (Figma или CSS продакшена):
1. Извлеки новые значения
2. Сравни с текущим файлом токенов (diff)
3. Обнови файл токенов
4. Проверь компоненты, использующие изменённые переменные
5. Обнови проектную директиву `design_system.md`

### 5.3 Обновление компонентов
Если компонент изменился в Figma:
1. Получи новый `get_design_context`
2. Сравни с текущей реализацией
3. Обнови код, сохранив обратную совместимость API
4. Обнови Code Connect маппинг если нужно

## Rules

### Что делать
- Всегда начинай с аудита (Фаза 1), даже если "всё ясно"
- Фиксируй всё в проектной директиве — это твоя память
- Используй токены, а не магические значения
- При создании компонентов — сверяйся со скриншотом из Figma
- Обновляй директиву при каждом значимом изменении

### Чего не делать
- Не хардкодь цвета, размеры, отступы — только токены
- Не создавай компоненты, которых нет в Figma DS
  (если нужен — сначала обсуди с пользователем)
- Не ломай существующий API компонентов при обновлении
- Не игнорируй шрифты — если кастомные, подключи
- Не путай сырой Figma-код (Tailwind) с кодом проекта —
  ВСЕГДА адаптируй к стеку

### Ограничения Figma Plugin API
- Локально установленные шрифты могут быть невидимы через
  `listAvailableFontsAsync()` — используй fallback
- `figma.currentPage` read-only — используй
  `figma.setCurrentPageAsync(page)`
- Код плагина ограничен ~15KB — разбивай большие операции
  на батчи
- Student plan: нет shared libraries — компоненты только
  внутри файла

## Output
- Проектная директива `design_system.md` (создаётся/обновляется)
- Файл токенов в формате проекта
- Компоненты в стеке проекта
- (Опционально) Code Connect маппинги
